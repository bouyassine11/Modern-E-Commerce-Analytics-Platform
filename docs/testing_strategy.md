# Testing Strategy

## Overview

Four-layer test pyramid covering unit, ETL, integration, and pipeline-level
testing. dbt provides its own test framework for data quality; Airflow DAG
validation tests run at the CI level.

```
         ┌──────────┐
         │ Pipeline │    Integration — full end-to-end (Docker Compose)
         │   E2E    │    scripts/integration_test.sh
        ┌┴──────────┴┐
        │  dbt Tests │    Data quality — 109 tests (singular + generic)
       ┌┴────────────┴┐
       │   ETL Tests   │    Transform logic, FK validation, dedup
      ┌┴───────────────┴┐
      │   Unit Tests     │    Generators, utils, monitoring, DAG import
      └──────────────────┘
```

---

## 1. Unit Tests

**Scope**: Individual functions in `etl/` — generators, transformers, utilities,
monitoring logger. No database required; all dependencies mocked.

**Framework**: pytest 8+ with pytest-cov.

| File | Tests | What it covers |
|---|---|---|
| `tests/test_generators.py` | 12 | Product count/price range, customer fields, order FK/dates, item totals, refund amount sign |
| `tests/test_transform.py` | 12 | Trim, lowercase, dedup, FK validation, per-table transforms, end-to-end pipeline |
| `tests/test_extract.py` | 4 | CSV reading, missing file, bad table name, extract_all |
| `tests/test_utils.py` | 3 | read_csv, write_csv, empty data |
| `tests/test_monitor.py` | 9 | Run lifecycle, task lifecycle, context manager, KPI snapshot, error handling, connection close |
| `tests/test_airflow_dag.py` | 7 | DAG import, task existence, downstream chain, no orphans, no cycles |

**Run**:
```bash
pytest tests/ -v -m "not integration"
```

**Coverage target**: ≥85% on `etl/` package.

---

## 2. ETL Tests

**Scope**: The full Extract → Transform → Load pipeline end-to-end, using a
real PostgreSQL instance (Docker-managed).

**Test data**: Small synthetic datasets (100 customers, 20 products, 200
orders) generated by `etl.generate` and written to `data/raw/`.

**Validation points**:

| Step | What is tested |
|---|---|
| Generate | Correct file count, row counts match config |
| Extract all | Files exist, parsable as CSV, correct column count |
| Transform | Row counts match (product of dedup + FK filtering), no orphan FKs |
| Load (dry-run) | Pipeline reports correct row counts without touching DB |
| Load (real) | `COPY FROM STDIN` completes, `raw.*_raw` tables populated |
| Monitoring | `monitoring.etl_runs` and `monitoring.etl_task_runs` populated |

**Run**:
```bash
./tests/integration/test_pipeline.sh          # Docker stack
./tests/integration/test_pipeline.sh --local  # local PostgreSQL
```

---

## 3. Integration Tests

**Scope**: Full stack validation against a live Docker Compose environment.
Tests the same flows as the Airflow DAG orchestrates.

**Trigger**: Manual or CI (scheduled).

**What it covers**:
- PostgreSQL health and schema creation
- ETL pipeline (generate → load)
- dbt (run → test → source freshness)
- Monitoring audit table population
- Report table queryability

**Run**:
```bash
docker compose up -d
./tests/integration/test_pipeline.sh
```

---

## 4. dbt Tests

**Scope**: Data quality validation at the database level — 109 tests across
the project.

**Categories**:
- **Singular tests** (8 files in `tests/`): Business-rule SQL assertions (row
  count parity, no null business keys, valid status values, positive prices,
  date recency, FK integrity, order total consistency, payment amount
  alignment)
- **Generic tests** (4 macros in `macros/`): Reusable `not_empty_string`,
  `valid_email`, `positive_value`, `date_in_past`
- **Column tests** (97 across `_stg__models.yml`, `_int__models.yml`,
  `_marts__models.yml`): `not_null`, `unique`, `accepted_values`,
  `relationships`, `positive_value`

**Run**:
```bash
dbt test --project-dir dbt/ecommerce
dbt test --project-dir dbt/ecommerce --select tag:core  # core subset
```

---

## 5. Airflow Tests

**Scope**: DAG structural correctness, task graph validity, import safety.

**What is tested**:
- DAG module compiles without errors (no missing imports, no runtime crashes
  at module level)
- All 11+ core tasks exist
- Task dependencies form a valid DAG (Airflow topological sort passes)
- No orphan tasks (every task reachable from start/end)
- Module-level `Variable.get()` calls are guarded against missing metadata DB

**Run**:
```bash
pytest tests/test_airflow_dag.py -v
```

## CI Integration

All of the above run automatically on every push to `main` and on every PR:

| CI Job | Tests executed | Environment |
|---|---|---|
| `lint` | ruff (Python), yamllint (YAML) | GitHub runner |
| `test` | Unit tests (pytest) | GitHub runner |
| `dbt-validate` | dbt debug + dbt compile | GitHub runner + Postgres service |
| `docker` | Build Airflow + Postgres images (no push) | GitHub runner + BuildKit |

Integration / E2E tests are **not** part of the default CI pipeline due to
Docker Compose overhead. They run:
- On-demand: `./tests/integration/test_pipeline.sh`
- Scheduled (nightly): via a separate GitHub Actions workflow

## Test Fixtures

Shared test data lives in `tests/conftest.py`:
- `sample_config`: Reduced generation config (100 customers, 20 products)
- `sample_products`: 4 rows with duplicates, nulls, and whitespace
- `sample_customers`: 3 rows with mixed casing and empty fields
- `sample_orders`: 3 rows (1 with orphan FK)
- `sample_order_items`: 5 rows (2 with orphan FKs)
- `sample_payments`: 3 rows (mixed case, refunds)

Temporary CSV files for extract tests are written to `tests/fixtures/`
(created on demand, cleaned up after each test).
